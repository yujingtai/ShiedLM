# ShieldLM 规则页统计增强设计

日期：2026-04-07

## 1. 目标

在现有规则页基础上补充更有解释性的统计信息，让页面不仅展示“当前有哪些规则”，还能够展示“这些规则在运行过程中命中了什么”。

本次增强的重点是提升规则页的展示完整度和分析价值，不改动核心防护链路。

## 2. 当前问题

当前规则页已经具备两部分内容：

- 基础运行指标
- 当前防护规则列表

但页面仍有两个明显不足：

1. 只能看到总量，无法看出攻击类型的分布情况。
2. 只能看到规则定义，无法看出哪些规则在实际运行中更常命中。

这会导致规则页更像“静态配置展示页”，而不是“带运行反馈的分析页”。

## 3. 本次范围

### 3.1 包含内容

- 新增攻击类型分布统计
- 新增规则命中排行
- 在规则页增加新的统计展示区块
- 为统计服务补充测试

### 3.2 不包含内容

- 图表库接入
- 时间趋势分析
- 规则在线增删改
- 规则命中详情钻取
- 导出统计结果

本次仍然坚持“小而完整”的范围控制，先让规则页具备更强的解释能力。

## 4. 方案比较

### 方案一：只补攻击类型分布

做法：

- 统计审计记录里不同攻击类型的出现次数
- 在规则页新增一个分布展示区

优点：

- 实现简单
- 能快速说明系统识别到了哪些攻击

缺点：

- 缺少和“规则本身”直接相关的反馈

### 方案二：攻击类型分布 + 风险等级占比

做法：

- 同时展示攻击类型分布和风险等级占比

优点：

- 能说明攻击类型，也能说明风险程度

缺点：

- 和现有基础指标重合度偏高
- 对“规则页”来说，规则反馈仍然不够强

### 方案三：规则命中排行 + 攻击类型分布

做法：

- 统计攻击类型分布
- 统计各规则命中次数，并按次数排序

优点：

- 一部分回答“识别到了什么”
- 一部分回答“哪些规则最常触发”
- 页面解释性最强，最适合当前规则页定位

缺点：

- 需要对审计记录中的攻击类型字段做一次解析汇总

## 5. 选定方案

本次采用方案三：规则命中排行 + 攻击类型分布。

原因：

- 规则页本身就应该更强调“规则效果”
- 相比只补总量，这种展示更容易讲清楚系统识别逻辑
- 不引入图表库的前提下，也能明显提升页面完成度

## 6. 设计说明

### 6.1 数据来源

统计仍然基于审计记录表。

原因：

- 审计记录已经是系统运行结果的统一落点
- 不需要额外新增统计表
- 对当前单用户演示版本来说足够

### 6.2 攻击类型分布

从 `AuditRecord.attackTypes` 字段中提取攻击类型，统计每种攻击类型出现次数。

展示要求：

- 按次数降序排列
- 展示攻击类型名称和命中次数
- 当没有数据时显示空状态

### 6.3 规则命中排行

本次不直接统计规则 key，而是基于当前已经持久化的 `attackTypes` 做“规则效果近似反馈”。

考虑到当前审计记录中保存的是攻击类型字符串，而不是完整规则 key 列表，本次规则命中排行采用“规则所属攻击类型出现次数”作为近似排名依据。

也就是说：

- 先拿到当前规则目录
- 再根据规则所属攻击类型，映射对应攻击类型的命中次数
- 最终按命中次数排序，形成规则页展示用排行

这样可以在不改审计表结构的前提下，先把规则页统计增强做出来。

### 6.4 服务层结构

`StatisticsService` 扩展为返回更丰富的汇总对象，除了现有四个基础指标外，再增加：

- 攻击类型分布列表
- 规则命中排行列表

为避免把复杂拼装逻辑塞进模板，服务层应提前完成：

- 字段解析
- 排序
- 展示对象转换

### 6.5 展示对象

建议新增两个轻量对象：

- `AttackTypeStatView`
- `RuleHitStatView`

用途：

- 减少模板内部拼接
- 让统计字段语义更清晰
- 后续继续扩展时更容易维护

### 6.6 页面层

`rules.html` 在现有结构基础上增加两个新板块：

1. 攻击类型分布
2. 规则命中排行

展示形式保持简洁，不引入图表库，优先使用：

- 指标卡片
- 简洁列表
- 表格式排行

这样既能和现有页面风格保持一致，也更适合当前项目阶段。

## 7. 测试策略

本次按 TDD 推进，优先补 `StatisticsServiceTests`。

重点覆盖以下行为：

1. 能正确聚合基础指标
2. 能把审计记录中的攻击类型拆分并统计次数
3. 能把规则目录映射为规则命中排行并按次数排序

控制器层只负责透传，可视实现复杂度决定是否追加测试，不强制扩大范围。

## 8. 风险与处理

### 风险 1：审计记录只保存攻击类型，不保存规则 key

处理方式：

- 本次先用“规则所属攻击类型命中次数”作为近似排行依据
- 后续如果要做更精确的规则级统计，再补持久化字段

### 风险 2：攻击类型字符串需要拆分，可能出现空值或格式不整齐

处理方式：

- 在服务层统一做 `split + trim + empty filter`
- 保证脏数据不会直接进入展示层

### 风险 3：页面信息过多导致视觉变乱

处理方式：

- 继续沿用现有卡片和表格风格
- 只新增两个紧凑区块，不做大面积重排

## 9. 预期结果

本次完成后，规则页应具备以下效果：

- 能展示系统识别到的攻击类型分布
- 能展示当前规则的命中排行
- 页面既有“规则定义”，也有“规则运行反馈”
- 在答辩或演示时，更容易解释系统规则层的实际作用
