# ShieldLM 真实模型接入设计

日期：2026-04-07

## 1. 目标

为 ShieldLM 接入兼容 OpenAI 协议的真实大模型服务，并将真实模型设为默认运行模式。

本次接入的重点不是扩展更多能力，而是把当前已经跑通的“输入检测 -> 风险处置 -> 模型生成 -> 输出审查”主链路真正接到外部模型上，让系统既能继续演示防护流程，也能体现真实调用能力。

## 2. 当前问题

当前项目虽然已经预留了 `ApiModelClient`，但仍存在三个明显缺口：

1. `api` 模式只是占位实现，尚未真正发起 HTTP 请求。
2. 配置层只有 `shieldlm.model.mode`，缺少真实模型所需的地址、模型名和密钥配置。
3. 项目默认仍走 `mock` 模式，不符合后续演示“默认连真实模型”的目标。

这意味着系统目前只能展示防护逻辑，还不能展示真实模型接入后的完整工程形态。

## 3. 本次范围

### 3.1 包含内容

- 实现兼容 OpenAI 的聊天补全调用
- 为模型接入补充配置对象
- 将默认模式切换为真实 API
- 支持通过环境变量注入 API key
- 为请求构造、响应解析和异常场景补充测试
- 更新项目运行说明

### 3.2 不包含内容

- 流式输出
- 多轮上下文记忆
- 工具调用或函数调用
- 图片、多模态输入
- 多厂商动态切换界面

本次仍然坚持“小而完整”的范围控制，先把真实模型接入这一条线稳定落地。

## 4. 方案比较

### 方案一：直接在现有 `ApiModelClient` 中使用原生 JDK `HttpClient`

做法：

- 保留当前 `ModelClient` 接口不变
- 新增配置类承接 `baseUrl`、`model`、`apiKey`、`chatPath`
- `ApiModelClient` 使用 `HttpClient` 发起 `POST /v1/chat/completions`
- 用 Jackson 解析标准 OpenAI 兼容响应

优点：

- 依赖最少
- 与当前项目结构贴合
- 改动集中，容易控制

缺点：

- 后续如果要扩成流式或多模态，还需要继续扩展

### 方案二：引入第三方 OpenAI SDK

做法：

- 通过 SDK 封装请求与响应
- 项目侧只处理参数与结果

优点：

- 上手快
- 某些协议细节由 SDK 处理

缺点：

- 对“兼容 OpenAI 但非官方地址”的适配弹性未必最好
- 增加外部依赖和学习成本
- 对当前项目规模来说偏重

### 方案三：用 Spring `RestClient` 或 `WebClient` 重写模型层

做法：

- 以 Spring 风格封装 HTTP 调用
- 后续便于扩展超时、拦截器和日志

优点：

- 和 Spring Boot 技术栈一致
- 可扩展性较好

缺点：

- 当前需求只是一条简单同步调用，收益不明显
- 相比原生 `HttpClient`，心智负担更高

## 5. 选定方案

本次采用方案一：在现有 `ApiModelClient` 中使用原生 JDK `HttpClient`。

原因：

- 当前只需要一个稳定、同步、兼容 OpenAI 的真实调用入口
- 不额外引入第三方 SDK，工程更轻
- 改动集中在模型层和配置层，不会影响现有防护链路

## 6. 设计说明

### 6.1 配置结构

新增模型配置项：

- `shieldlm.model.mode`
- `shieldlm.model.base-url`
- `shieldlm.model.model-name`
- `shieldlm.model.api-key`
- `shieldlm.model.chat-path`

默认值要求：

- `mode` 默认 `api`
- `base-url` 默认 `https://ai.td.ee`
- `model-name` 默认 `gpt5.4`
- `chat-path` 默认 `/v1/chat/completions`

其中 `api-key` 不写入仓库，统一通过环境变量注入。

### 6.2 运行方式

系统默认走真实模型模式。

也就是说，正常启动时会优先创建 `ApiModelClient`。如果本地没有配置 API key，则在启动阶段明确失败，直接提示缺失配置，而不是启动后再在页面里暴露模糊错误。

仍保留 `mock` 模式，但只作为手动应急开关，不再作为默认值。

### 6.3 请求格式

按照兼容 OpenAI 的标准聊天补全协议发送请求：

- 方法：`POST`
- 路径：`{baseUrl}{chatPath}`
- 请求头：
  - `Authorization: Bearer <apiKey>`
  - `Content-Type: application/json`
- 请求体核心字段：
  - `model`
  - `messages`

由于当前 ShieldLM 的模型输入仍是单条字符串，本次先将处理后的提示词包装成一条 `user` 消息发送，不额外引入系统消息模板。

### 6.4 响应解析

优先从标准兼容格式中提取：

- `choices[0].message.content`

如果响应中不存在有效内容，则抛出明确异常，提示“模型返回格式不符合预期”，避免静默返回空字符串。

### 6.5 异常处理

本次只做最小但明确的异常策略：

- 缺少 API key：启动阶段失败
- HTTP 非 2xx：抛出包含状态码和响应体摘要的异常
- 响应结构缺失：抛出解析异常
- 网络异常：抛出模型调用失败异常

这里不做自动重试，避免在答辩演示时把真实错误隐藏掉。

### 6.6 对现有链路的影响

`ShieldPipelineService` 不需要调整核心流程，继续通过 `ModelClient` 抽象调用模型。

这样可以保证：

- 输入检测逻辑不变
- 输出审查逻辑不变
- 审计记录逻辑不变
- 页面展示逻辑不变

本次只是在模型适配层把“模拟返回”升级为“真实请求”。

## 7. 测试策略

本次按 TDD 推进，优先补模型层测试，再落生产代码。

重点覆盖：

1. 能按兼容 OpenAI 协议发出请求
2. 能正确携带 `Authorization` 与 `model`
3. 能从 `choices[0].message.content` 解析出回复
4. 当响应缺失内容或状态码异常时，能抛出明确异常

测试不依赖外部真实网络，优先使用本地轻量 HTTP 服务模拟响应。

## 8. 风险与处理

### 风险 1：私有兼容接口字段与标准协议存在轻微差异

处理方式：

- 先按最常见的 OpenAI 兼容格式实现
- 保持 `baseUrl` 与 `chatPath` 可配置
- 通过明确异常快速暴露差异点，便于后续调整

### 风险 2：默认真实模型可能导致未配置密钥时无法启动

处理方式：

- 在 README 和配置中明确写出环境变量要求
- 启动阶段直接报错，避免误判为系统本身异常

### 风险 3：真实模型输出不稳定，影响演示节奏

处理方式：

- 保留 `mock` 模式作为兜底
- 演示前优先验证接口连通性
- 真实模型只替换模型层，不影响整体防护逻辑

## 9. 预期结果

本次完成后，项目应具备以下效果：

- 默认启动即连接真实大模型
- 防护主链路仍然完整可用
- 当输入被放行或改写后，可以拿到真实模型回复
- 当真实模型返回敏感内容时，输出审查仍能进行兜底处理
- 项目既能演示安全流程，也能体现真实模型接入能力
