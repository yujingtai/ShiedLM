# ShieldLM 审计检索与分页设计

日期：2026-04-07

## 1. 目标

在现有审计日志页基础上补充两个能力：

- 支持按关键词检索审计记录
- 支持按页查看审计记录

这次增强不改变 ShieldLM 的核心防护链路，重点是提升审计页的可用性与展示完整度，让页面从“基础展示”升级为“可检索、可浏览的审计台”。

## 2. 当前问题

当前审计页虽然已经支持按风险等级和防御动作筛选，但仍有两个明显短板：

1. 所有记录会一次性加载并展示，数据量增大后页面可读性会下降。
2. 无法按关键词快速定位某一类提示词、攻击类型或最终回复。

这两个问题不会影响 MVP 主链路运行，但会影响后续演示时的操作效率，也会削弱系统“可审计、可复盘”的完整性。

## 3. 本次范围

### 3.1 包含内容

- 审计页新增关键词输入框
- 审计查询支持 `keyword + riskLevel + defenseAction` 组合筛选
- 审计结果支持分页展示
- 页面展示当前页、总页数和结果数量摘要
- 保留现有统计摘要区
- 为新增能力补充控制器测试

### 3.2 不包含内容

- CSV 导出
- 审计详情弹窗
- 高级排序方式切换
- 多字段高亮匹配
- 审计页单独图表分析

这样可以把这一批控制在一个小而完整的范围内，先把最核心的检索体验补齐。

## 4. 方案比较

### 方案一：数据库查询分页 + 关键词搜索

做法：

- 在仓储层直接完成筛选与分页
- 服务层返回分页结果
- 控制器和模板只负责参数绑定与页面展示

优点：

- 结构清晰
- 后续数据量增大时更稳定
- 符合真实系统的处理方式

缺点：

- 相比内存筛选，多改几层代码

### 方案二：内存筛选 + 手动分页

做法：

- 仍旧先把全部审计记录取出
- 在 Java 层内存过滤并切分列表

优点：

- 改动小

缺点：

- 数据多时性能和结构都不理想
- 后续大概率还要再重做

### 方案三：只补关键词搜索，不做分页

做法：

- 先补关键词筛选，分页延后

优点：

- 最省时间

缺点：

- 提升不完整
- 审计页成品感不够强

## 5. 选定方案

本次采用方案一：数据库查询分页 + 关键词搜索。

原因：

- 当前项目已经使用 Spring Data JPA，继续沿用分页查询成本较低
- 这套实现方式更适合后续继续扩展导出、排序和统计能力
- 对答辩演示来说，分页和搜索一起落地，完成度会更高

## 6. 设计说明

### 6.1 查询参数

审计页保留并扩展以下请求参数：

- `keyword`：关键词检索，可匹配 `userPrompt`、`attackTypes`、`finalReply`
- `riskLevel`：风险等级筛选
- `defenseAction`：防御动作筛选
- `page`：页码，从 0 开始传递，页面展示时按 1 开始显示

页面上的筛选、搜索和分页应保持联动，也就是翻页时仍保留当前筛选条件。

### 6.2 仓储层

`AuditRecordRepository` 新增分页查询接口，按 `createdAt desc` 返回结果。

查询逻辑要求：

- 当 `keyword` 为空时，不影响结果
- 当 `riskLevel` 为空时，不限制风险等级
- 当 `defenseAction` 为空时，不限制防御动作
- 关键词匹配使用包含关系即可，不做分词或模糊高亮

### 6.3 服务层

`AuditLogService` 从返回 `List<AuditRecord>` 升级为返回 `Page<AuditRecord>`。

职责：

- 统一封装查询参数
- 统一页大小配置
- 继续保留保存审计记录的方法

为避免页面过长，本次页大小固定为一个较小值，适合演示即可。

### 6.4 控制器层

`AuditController` 新增对 `keyword` 和 `page` 的处理。

页面模型中新增以下信息：

- 当前页记录列表
- 当前关键词
- 当前页码
- 总页数
- 是否有上一页
- 是否有下一页
- 当前结果数量摘要

现有摘要区继续基于“当前筛选后的记录”展示，不扩展到“全量数据统计”。

### 6.5 页面层

`audit.html` 做三处增强：

1. 在筛选表单中加入关键词输入框  
2. 在表格下方加入分页导航  
3. 在摘要区加入当前页结果提示

界面保持现有风格，不做大改版，只做与功能直接相关的增量调整。

## 7. 测试策略

本次按 TDD 推进，先补控制器测试，再做实现。

优先覆盖以下场景：

1. 访问审计页时返回分页结果与分页状态
2. 带 `keyword` 参数访问时，控制器会把关键词和分页结果传入模型
3. 关键词、风险等级、防御动作组合查询时，服务层调用参数正确

如果控制器测试通过，再补充仓储或服务层测试视实现复杂度决定，不额外扩大范围。

## 8. 风险与处理

### 风险 1：分页参数与页面展示页码不一致

处理方式：

- 后端继续使用 0 基页码
- 页面统一显示为 1 基页码
- 在模板中显式区分“请求值”和“展示值”

### 风险 2：关键词为空时查询条件失效

处理方式：

- 在服务层对空字符串做标准化处理
- 保证仓储查询在空关键词时仍可正常返回筛选结果

### 风险 3：模板参数变多后页面判断复杂

处理方式：

- 控制器层明确提供 `hasPrevious`、`hasNext`、`pageNumberDisplay` 等字段
- 尽量减少模板内部复杂表达式

## 9. 预期结果

本次完成后，审计页应具备以下效果：

- 可以通过关键词快速定位相关审计记录
- 可以分多页查看记录，不再一次性展示全部内容
- 可以保留筛选条件进行翻页
- 页面整体展示更接近一个完整的审计管理页面

## 10. 后续衔接

这一批完成后，审计页下一步最适合继续补的能力有两项：

- 导出当前筛选结果
- 增加更细的统计视图或规则命中分布

但这两项不进入本次实现范围。
