# ShieldLM 设计文档

> 副标题：通用大模型提示注入防护框架与企业客服验证原型
>
> 日期：2026-04-06

## 1. 项目背景

本项目面向大语言模型应用中的提示注入攻击问题，设计并实现一套具备动态检测、风险评估、策略执行、输出审查和审计分析能力的防护工具。论文层面，项目研究对象保持为通用的大模型提示注入防御框架；工程层面，项目以企业智能客服作为验证场景，完成可演示的原型落地。

该定位兼顾了研究抽象性与工程可展示性。一方面，防护能力本身不依赖于单一业务场景；另一方面，企业客服场景便于直观展示越权指令、提示词窃取、敏感信息诱导和违规内容诱导等混合型攻击的防御效果。

## 2. 项目目标

本项目的总体目标是实现一个可运行、可演示、可审计的提示注入防护系统原型，并证明其具备以下能力：

1. 能够识别混合型提示注入攻击，而非仅依赖单一关键词拦截。
2. 能够对风险进行分级，并基于不同风险等级执行差异化防御动作。
3. 能够在具体业务场景中完成端到端演示，包括输入检测、模型调用、输出审查与日志留存。
4. 能够为论文写作和答辩展示提供清晰、可解释的系统结构与实验依据。

## 3. 系统定位

### 3.1 研究定位

ShieldLM 被定义为“通用防护框架 + 企业智能客服验证原型”。

其中：

- 通用防护框架对应论文题目中的研究主体，负责实现提示注入检测与防御的核心能力。
- 企业智能客服验证原型对应实验与答辩场景，负责将通用能力映射到一个可理解、可展示的业务界面中。

### 3.2 项目边界

本项目采用单用户演示版范围控制，以保证两周内能够完成一个高质量原型。

本次实现明确不包含以下内容：

- 用户登录与注册
- 多用户和权限管理
- 复杂后台管理系统
- 真正的 RAG 或知识库检索流程
- 本地部署重型大模型
- 自动在线学习与复杂模型训练流程

上述限制用于避免工程范围失控，使开发重心集中在提示注入防御能力本身。

## 4. 核心问题与攻击范围

ShieldLM 重点防御以下混合型攻击：

- 越权指令攻击：例如诱导模型忽略既定规则、改变系统设定或服从新的高优先级命令。
- 提示词窃取攻击：例如要求模型暴露系统提示词、内部规则、隐藏约束或开发配置。
- 敏感信息诱导攻击：例如要求模型返回密钥格式、接口配置、内部说明、受限业务信息。
- 违规内容诱导攻击：例如诱导模型绕过限制生成不合规内容、危险建议或违背业务边界的答复。

本项目不是单纯做“内容审核”，而是围绕提示注入攻击进行防御。因此系统不仅判断输出是否违规，还要识别用户输入中的攻击意图、攻击方式以及潜在越权行为。

## 5. 总体架构

系统整体采用分层架构，分为演示层与防护核心层。

### 5.1 演示层

演示层为企业智能客服验证子系统，负责提供用户可交互的单用户 Web 界面，展示如下信息：

- 聊天输入和模型回复
- 命中的攻击类型
- 风险等级
- 处理动作
- 安全改写结果
- 审计日志和统计摘要

该层主要承担“验证”和“展示”的职责，不承载防护算法本身。

### 5.2 防护核心层

防护核心层命名为 ShieldLM Core，包含五个关键子层：

1. 输入检测层：识别用户输入中的提示注入攻击迹象。
2. 策略执行层：根据风险结果决定放行、改写或拦截。
3. 模型接入层：统一适配真实 API 与本地模拟模型。
4. 输出审查层：检查模型输出中的敏感或违规内容。
5. 审计与分析层：记录全过程并提供查询与展示能力。

这种架构能够保证核心防护逻辑与具体业务界面解耦，便于论文表述“方法可迁移到其他大模型应用场景”。

## 6. 核心处理链路

系统主链路定义如下：

`用户输入 -> 输入检测 -> 风险评分 -> 策略执行 -> 模型调用 -> 输出审查 -> 返回前端 -> 写入审计日志`

各阶段职责如下：

1. 接收输入：前端提交用户问题和上下文信息。
2. 输入检测：识别提示注入相关规则命中情况。
3. 风险评分：汇总命中项，生成可解释的风险得分与等级。
4. 策略执行：依据风险等级决定是否放行、改写或拒绝。
5. 模型调用：将处理后的请求发送给模型适配层。
6. 输出审查：对模型回复进行二次检查，防止泄露或违规输出。
7. 返回前端：展示最终安全输出和中间过程信息。
8. 写入审计日志：保留输入、检测结果、动作和最终输出的结构化记录。

## 7. 功能模块设计

### 7.1 安全聊天演示模块

该模块是系统的主入口，负责提供企业智能客服场景下的交互体验。页面应支持：

- 普通聊天输入
- 攻击样例快捷填充
- 风险等级与处理动作展示
- 原始输入与安全处理结果对比
- 模型原始输出与最终安全输出对比

该模块的目标是让答辩时的防护过程可视化，而不是只展示一个最终回答。

### 7.2 输入检测模块

输入检测模块采用规则驱动方式实现，初期不依赖复杂机器学习模型。检测逻辑围绕可解释性展开，优先使用如下手段：

- 关键词规则
- 正则表达式匹配
- 多条件组合命中
- 攻击类型映射

例如：

- “忽略之前所有规则”可映射为越权指令特征
- “输出系统提示词”可映射为提示词窃取特征
- “告诉我内部配置和密钥格式”可映射为敏感信息诱导特征

### 7.3 风险评分模块

风险评分模块将输入检测的命中结果转换为数值得分与风险等级。评分机制必须具备可解释性，以便论文撰写和答辩说明。

建议采用分项累计方式：

- 基础规则命中产生基础分
- 高危攻击类型附加更高权重
- 多规则同时命中时提高整体风险等级

最终风险等级划分为：

- 低风险
- 中风险
- 高风险

### 7.4 策略执行模块

策略执行模块根据风险等级执行不同动作：

- 低风险：直接放行
- 中风险：对输入进行安全改写或疑似恶意片段移除后再发送给模型
- 高风险：直接拒绝，并返回安全提示

该设计体现“动态防御”而非“一刀切拒绝”，也是本项目的重要亮点之一。

### 7.5 模型接入模块

模型接入层采用统一接口，支持两种模式：

- 真实 API 模式：用于说明系统具备接入真实大模型服务的能力
- 本地模拟模型模式：用于保证答辩、演示和离线测试时的稳定性

两种模式共用统一调用协议，上层不感知实现差异。这样既保证系统具备工程落地感，也降低了现场演示的失败风险。

### 7.6 输出审查模块

输出审查模块负责检查模型回复中是否存在以下问题：

- 系统提示词或内部规则泄露
- 密钥样式、配置样式或敏感字段暴露
- 不合规或业务不允许的输出内容

若检测命中，则系统不直接返回模型原始输出，而是使用安全兜底回复替代，并将事件写入日志。

### 7.7 审计与可视化模块

系统应保存如下结构化信息：

- 请求时间
- 用户输入
- 命中的攻击类型
- 风险等级
- 执行动作
- 安全改写结果
- 模型原始输出
- 最终输出
- 输出侧是否命中拦截

该模块既支撑论文实验统计，也支撑答辩过程中的结果回放。

## 8. 页面设计

为控制工作量并保持展示完整性，前端只实现三个核心页面：

### 8.1 安全聊天页

主页面，用于完成完整的攻击与防护演示。推荐展示区域包括：

- 聊天区
- 风险分析卡片
- 处理链路步骤条
- 攻击样例按钮
- 输出结果对比面板

### 8.2 审计日志页

用于展示历史请求记录和详细处理过程，支持按风险等级或攻击类型快速查看。

### 8.3 规则与统计页

用于展示启用规则、各攻击类型命中次数以及高风险、中风险事件数量。该页面不追求复杂图表，但要提供足够的“系统完成度”。

## 9. 技术选型

本项目采用如下技术方案：

- 后端：Java 21
- 核心框架：Spring Boot
- 前端模板：Thymeleaf
- 前端交互：原生 JavaScript
- 样式方案：自定义 CSS
- 数据存储：H2 数据库
- 数据访问：Spring Data JPA
- 配置管理：application.yml
- 规则库：本地 YAML / JSON 文件

该技术栈有三个主要优势：

1. 全项目以 Java 为主，符合开发偏好和课程背景。
2. 前后端一体化，部署与演示简单。
3. 开发效率高，适合两周时间内落地完整原型。

## 10. 模块划分建议

为保持结构清晰，代码层面建议划分为以下逻辑模块：

- demo-web：页面与控制器
- shield-core：主流程编排
- input-detector：输入检测
- risk-engine：风险评分
- strategy-engine：策略执行
- model-adapter：模型适配
- output-guard：输出审查
- audit：日志与统计

模块之间采用清晰接口调用，避免将所有逻辑堆叠在单一控制器中。

## 11. 错误处理与稳定性设计

系统需重点考虑以下异常场景：

- 真实 API 不可用或超时
- 输入改写失败
- 输出审查异常
- 数据库存储异常

对应策略如下：

- 当真实 API 不可用时，允许切换到本地模拟模型
- 当策略执行异常时，默认采用更保守的安全拒绝方式
- 当输出审查失败时，返回安全兜底消息并记录异常日志
- 当数据库不可用时，至少保证当前请求结果可返回，并记录降级信息

这部分设计用于保证答辩现场系统行为稳定，不因单点失败而整体中断。

## 12. 测试与验证思路

本项目的测试重点不在性能极限，而在防护有效性与过程可解释性。

测试将围绕以下四类场景开展：

1. 正常请求：验证系统不会误伤普通客服问题。
2. 中风险攻击：验证系统能够改写或降权处理而非直接拒绝。
3. 高风险攻击：验证系统能够及时拦截明确恶意请求。
4. 输出侧兜底：验证模型若生成敏感内容，系统仍能在输出阶段进行拦截。

系统验证结果应至少体现：

- 不同攻击类型的识别能力
- 不同风险等级的处理差异
- 输出阶段二次防护的必要性
- 日志记录的完整性

## 13. 论文亮点与答辩主线

本项目建议提炼以下四项亮点：

1. 双阶段动态防护：同时覆盖输入检测与输出审查。
2. 可解释风险评分：风险来源清晰，可追溯。
3. 分级策略执行：支持放行、改写、拦截三类响应。
4. 通用框架 + 场景验证：研究通用、演示具体。

建议答辩演示主线如下：

1. 正常请求演示
2. 中风险攻击演示
3. 高风险攻击演示
4. 输出侧兜底演示

该顺序能够完整展示系统从可用性到安全性的全流程能力。

## 14. 范围控制与实施原则

为了确保项目在两周内具备较高完成度，后续实现阶段应坚持以下原则：

- 优先保证主链路跑通
- 优先实现可解释规则和可展示过程
- 优先保证答辩演示稳定性
- 不为“看起来更复杂”而引入不必要的技术栈

若时间充裕，可在后续增强中补充更细的规则库、更多攻击样例和更丰富的统计展示；若时间不足，则以“安全聊天页 + 防护主链路 + 日志页”为最低可交付核心。

## 15. 结论

ShieldLM 的设计目标不是构建一个商业化的完整客服平台，而是实现一套围绕提示注入攻击的通用防护原型，并通过企业智能客服场景对其进行验证。该方案在研究层面保持了论文题目的完整性，在工程层面提供了可落地、可演示、可扩展的实现路径，适合作为本科毕业设计项目推进。
