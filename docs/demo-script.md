# ShieldLM 演示脚本

本文档用于统一演示流程，尽量让展示过程稳定、节奏清晰。

## 演示前检查

1. 启动项目并确认首页可以访问
2. 打开聊天页、审计页、规则页三个入口
3. 确认数据库中已有少量演示记录，或者准备现场重新跑一遍样例

## 演示主线

### 第一步：先说明系统定位

说明重点：

- 项目核心不是做一个完整客服平台，而是验证面向大模型的提示注入防护链路
- 当前用客服场景做工程化原型承载，便于把风险识别和防御过程展示出来

### 第二步：展示正常请求

操作建议：

- 在首页输入正常退款咨询问题
- 提交后展示风险等级、处理动作和最终回复

说明重点：

- 正常请求不会被误伤
- 系统不仅要“能拦”，也要“能用”

### 第三步：展示中风险请求

操作建议：

- 使用“规则绕过尝试”样例
- 重点展示处理后提示词和命中规则明细

说明重点：

- 系统不是只做关键词拦截
- 中风险场景采用改写和降权处理，而不是全部一刀切拒绝

### 第四步：展示高风险混合攻击

操作建议：

- 使用“混合型提示注入攻击”样例
- 提交后展示高风险判定与直接拦截

说明重点：

- 输入中同时包含越权与提示词窃取意图时，会被识别为高风险
- 系统会在模型调用前终止请求

### 第五步：展示输出兜底

操作建议：

- 使用“输出侧兜底”样例
- 观察原始回复与最终安全回复的差异

说明重点：

- 仅做输入检测还不够
- 输出阶段仍需要二次审查，防止敏感信息泄露

### 第六步：切到审计页和规则页

操作建议：

- 在审计页展示刚才几次请求记录
- 在规则页展示当前规则与统计摘要

说明重点：

- 系统不仅能拦截，还能留痕与复盘
- 审计记录和统计信息可以支撑实验分析与结果说明

## 演示时应重点强调的点

- 输入检测与输出审查是双阶段防护
- 风险评分和防御动作具备可解释性
- 当前实现强调原型稳定性与可展示性
- 系统架构保留了继续扩展真实模型接入和规则管理的空间
