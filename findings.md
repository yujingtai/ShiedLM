# Findings & Decisions

## Requirements
- 项目必须围绕“面向大模型的提示注入攻击”这一论文主题展开，不能被收缩成单一业务系统
- 需要有可运行、可演示、可讲解的工程原型，方便开题和答辩时展示
- 采用单用户演示模式即可，不引入登录、权限或多用户协作
- 后端使用 Java，其余技术栈以本项目落地效率和演示稳定性为优先
- 开发过程保留连续的 Git 记录，尽量按小功能粒度提交
- 代码可以有注释，但注释必须中性、技术化，避免明显的“解释给别人看”的 AI 口吻

## Product Positioning
- ShieldLM 的定位是“提示注入攻击防护演示原型”，而不是完整的企业安全平台
- 项目要能从输入攻击、检测命中、策略处置、输出防护、审计留痕一路讲通
- 展示重点不是业务流程，而是提示注入防护链路如何运行、如何记录、如何分析

## Technical Decisions
| Decision | Rationale |
|----------|-----------|
| 使用 Spring Boot + Thymeleaf + H2 | 部署和演示成本低，适合两周左右完成可运行原型 |
| 使用 MockModelClient 作为默认模型客户端 | 不依赖外部大模型接口，避免答辩时网络和额度带来的不确定性 |
| 审计能力基于数据库记录做聚合展示 | 逻辑直观，方便快速形成可解释的页面数据 |
| 审计页使用数据库查询分页 + 关键词搜索 | 比内存过滤更贴近真实实现，也更利于解释系统扩展性 |
| 规则页增加攻击类型分布与规则命中排行 | 让规则页不只展示“定义了什么规则”，还能展示“规则近期反映了什么风险态势” |
| 规则命中排行先采用近似方案 | 当前不新增命中明细表结构，直接复用 `AuditRecord.attackTypes` 做聚合映射 |

## Research Findings
- 当前 MVP 的三块核心页面已经形成闭环：聊天演示页、审计日志页、规则与统计页
- 审计页已具备检索、筛选、分页能力，能够支持答辩时展示“风险留痕”和“可追溯”
- 规则页现在既能展示规则目录，也能展示运行态统计，更适合作为答辩讲解页
- 现阶段最适合继续推进的方向，是继续完善演示稳定性和论文支撑材料，而不是盲目扩展业务功能

## Issues Encountered
| Issue | Resolution |
|-------|------------|
| 早期部分中文文件存在编码混乱 | 逐步重写相关文档和模板，统一为可直接阅读的中文内容 |
| `RulesControllerTests` 里对 Java record 使用 `hasProperty` 断言失败 | 改为基于页面实际输出内容做断言，更贴近真实页面行为 |
| PowerShell 中查看部分中文内容会出现乱码 | 按 UTF-8 重新读取文件内容，避免误判为源码损坏 |
| Mockito 在 JDK 21 下提示动态 agent 警告 | 当前不影响测试通过，后续如需收口可再处理测试配置 |

- 已完成：规则页基础指标、攻击类型分布、规则命中排行、规则目录展示
